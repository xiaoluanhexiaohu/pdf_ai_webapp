# PDF AI 智能插图系统（中文教程）

这是一个可以在本地运行的 FastAPI 网页项目。

## 零、快速试跑（先验证项目没问题）

项目里额外附带了一个 `sample_assets/` 目录，里面有：

- `demo.pdf`：示例 PDF；
- `img1.png`、`img2.png`：示例图片。

你可以先启动网页，再用这些文件测试一遍完整流程。
这一步的目的不是测试 AI 准确率，而是先验证：

- 文件上传是否正常；
- 锚点是否能被找到；
- PDF 是否能成功输出。

建议你在锚点输入框中先填：

```text
SITE PHOTO:
PRODUCT INFO:
```

这样你能最快看到项目先跑通。

---

功能链路：

1. 上传一个 PDF。
2. 上传多张图片。
3. 输入 PDF 里用于定位的固定文字锚点。
4. 程序先解析 PDF，找出这些锚点在第几页、什么坐标。
5. 如果你配置了 OpenAI API Key，就把“锚点信息 + 页面截图 + 用户图片”一起发给模型，让 AI 判断哪张图该放到哪个锚点旁边。
6. 程序根据页面剩余空间决定把图放在锚点下方、右侧，或者追加一个补充页。
7. 输出新的 PDF。

---

## 一、适合什么场景

适合：

- 合同、报告、方案书、项目材料里有固定标题或固定提示词；
- 你希望系统根据图片内容自动判断它应该跟哪一段文字对应；
- 你接受第一版是“固定版面坐标落图”，而不是 Word 那种自动整篇重排版。

不适合：

- 需要像 Word 一样插图后全文自动下移、跨页重排；
- PDF 是纯扫描件、而且你还没做 OCR；
- 业务规则极其复杂，但没有给系统明确锚点。

## 三、开发思路，先理解这 4 个核心层

### 1. PDF 定位层

这个层只做一件事：
**找到锚点文字在 PDF 里的坐标。**

例如你在表单里输入：

```text
现场照片：
产品说明：
附件图：
```

程序会遍历 PDF 的所有页面，对每个锚点调用 `page.search_for()`，拿到坐标矩形。

一旦找到锚点，就会得到类似这样的信息：

```json
{
  "anchor_id": "p2_现场照片：_1",
  "page_number": 2,
  "anchor_text": "现场照片：",
  "anchor_rect": [120.0, 355.0, 210.0, 372.0]
}
```

这一步非常重要，因为后面所有“放图位置”的计算，都依赖这个坐标。

---

### 2. AI 匹配层

这一层只做“语义决策”，不做像素级落点。

AI 的输入包括：

- 用户上传的图片；
- 每个锚点的文字内容；
- 锚点上下文；
- 对应页面截图。

AI 的输出包括：

- 这张图片应该匹配哪个 `anchor_id`；
- 建议放在 `below`、`right` 还是 `appendix_page`；
- 给一个置信度和简短原因。

注意：
**不要让 AI 直接报 x/y 坐标。**

坐标应该由程序来算，因为程序更稳定、更可控。

---

### 3. 排版引擎层

这一层会根据锚点矩形和页面大小计算候选位置。

现在已经做了 3 种模式：

- `below`：放在锚点下方；
- `right`：放在锚点右侧；
- `appendix_page`：本页放不下时，自动在文档后追加一页。

同时会做简单的碰撞检测：

- 已有文字块不能覆盖；
- 已有图片区域不能覆盖；
- 不能超出页边距。

如果 AI 推荐 `below`，但本页空间不够，程序会自动改成 `right` 或 `appendix_page`。

---

### 4. PDF 写回层

这一步最简单：

- 打开原 PDF；
- 找到目标页；
- 用 `page.insert_image()` 把图插进去；
- 如果需要补充页，就 `doc.new_page()`；
- 保存新 PDF。

## 四、如何在 PyCharm 里一步一步跑起来

### 第 1 步：新建项目

在 PyCharm 里：

1. 选择 `Open`，打开本项目目录；
2. 确认 Python 解释器是 3.11 或以上；
3. 打开底部 Terminal。

### 第 2 步：创建虚拟环境

Windows PowerShell：

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
```

Windows CMD：

```cmd
python -m venv .venv
.venv\Scripts\activate
```

macOS / Linux：

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 第 3 步：安装依赖

```bash
pip install -r requirements.txt
```

### 第 4 步：配置环境变量

把 `.env.example` 复制成 `.env`：

```bash
cp .env.example .env
```

Windows 可手动复制文件。

然后编辑 `.env`：

```env
OPENAI_API_KEY=你的key
OPENAI_MODEL=gpt-4.1-mini
```

如果你暂时没有 key，也可以先留空。
项目仍然能跑，只是会改用“顺序匹配锚点”的兜底逻辑。

### 第 5 步：启动项目

```bash
uvicorn app.main:app --reload
```

看到类似输出后，说明成功：

```text
Uvicorn running on http://127.0.0.1:8000
```

浏览器打开：

```text
http://127.0.0.1:8000
```

## 五、网页怎么使用

首页有 3 个输入：

1. 上传 PDF；
2. 上传多张图片；
3. 输入锚点文字（每行一个）。

例如锚点：

```text
现场照片：
产品说明：
附件图：
```

点击“开始处理”后：

- 程序会解析 PDF；
- 找锚点；
- 渲染锚点所在页截图；
- 调 AI 做匹配；
- 计算坐标；
- 写回 PDF；
- 页面展示匹配结果并提供下载。

## 六、重点代码解释

### 1. `pdf_parser.py`

这里最关键的是：

- `find_anchor_candidates()`：查找所有锚点；
- `render_page_to_image()`：把页面渲染成 PNG，提供给 AI 看；
- `get_page_occupied_rects()`：拿到文字块和已有图片区域，用于碰撞检测。

你以后如果要支持 OCR，也优先从这里扩展。

---

### 2. `ai_matcher.py`

这里分成两种路径：

#### 路径 A：配置了 OpenAI key

就把以下内容发给模型：

- 结构化 anchors JSON；
- 用户图片；
- PDF 页面截图；
- 规则说明。

希望模型返回 JSON：

```json
{
  "placements": [
    {
      "image_id": "img_1",
      "filename": "a.png",
      "anchor_id": "p2_现场照片：_1",
      "page_number": 2,
      "mode": "below",
      "size_hint": "medium",
      "confidence": 0.91,
      "reason": "图片内容与该锚点语义更接近"
    }
  ]
}
```

#### 路径 B：没有 key 或 AI 出错

程序会自动走 `_heuristic_match()`：

- 第 1 张图配第 1 个锚点；
- 第 2 张图配第 2 个锚点；
- 不够就循环。

这样做的目的不是最终智能，而是为了保证项目第一天就能跑通。

---

### 3. `layout_engine.py`

这是整个项目里最值得你反复看的文件。

它的职责是：

- 根据页面大小限制图片最大宽高；
- 先尝试 `below`；
- 再尝试 `right`；
- 如果两个都放不下，就返回 `appendix_page`；
- 插图位置不能和已有文字块、已有图片重叠。

你后续可以在这里继续升级：

- 支持“居中插图”；
- 支持“下方多图宫格”；
- 支持“根据锚点周围空白区域自动搜索最佳矩形”。

---

### 4. `pdf_writer.py`

这个文件目前做了两种写法：

- 插到原页；
- 追加补充页。

追加补充页时，会在页顶写：

- 标题：`AI 自动补充图片页`
- 关联页码和锚点文字。

这个设计很适合第一版，因为很多 PDF 本页其实没有足够空间。

## 七、你接下来最值得做的 6 个升级

### 升级 1：加入 OCR

现在项目默认更适合“文字型 PDF”。
如果你常处理扫描件，可以加一层 OCR：

- 先把扫描 PDF OCR 成可搜索 PDF；
- 再走现在这套锚点查找流程。

### 升级 2：做人工确认页

现在 AI 匹配完成后，直接生成 PDF。
更实用的做法是：

- 先展示“图片 -> 锚点”的建议结果；
- 用户可手动调整；
- 再点击“确认生成 PDF”。

### 升级 3：支持图片说明文字

你可以在上传图片时增加一列“图片备注”，例如：

- 现场全景
- 门头照片
- 施工设备

然后把这些说明也一起发给 AI，匹配准确率会更高。

### 升级 4：支持批量处理多个 PDF

当前是一份 PDF。
你以后可以改成：

- 左侧上传多个 PDF；
- 右侧上传图片包；
- 按文件名规则自动分配。

### 升级 5：增加日志与调试面板

建议把这些东西保存下来：

- 找到的 anchors；
- 页面截图；
- 发给 AI 的 prompt；
- AI 返回结果；
- 最终落图坐标。

这样调错会非常快。

### 升级 6：接数据库

如果你以后要做成多人使用的正式系统，可以接：

- SQLite：本地原型够用；
- PostgreSQL：生产环境更稳。

数据库里可以存：

- 任务记录；
- 结果 PDF 路径；
- 用户上传历史；
- 模型调用日志。

## 九、目前这版的边界

你要清楚，当前版本是“能跑、结构正确、方便继续迭代”的第一版，不是终极版。

目前边界：

- 没有真正 OCR 流程；
- 没有人工确认界面；
- 没有数据库；
- 没有用户系统；
- AI 返回 JSON 失败时，会走兜底规则；
- 对复杂版面只做了基础碰撞检测。

但是这些都不影响你用它做第一版项目和继续开发。

## 十、最重要的理解

你这个项目不是“AI 自动编辑 PDF”的一句话功能，而是下面 4 件事拼起来：

1. **用程序找锚点**；
2. **用 AI 做语义匹配**；
3. **用程序算落图坐标**；
4. **用程序写回 PDF**。

把这 4 层分清楚，你后面加什么功能都不会乱。

如果你后续继续做，我最建议你下一步升级的是：

**加一个“AI 建议结果确认页”，让用户可以手动改匹配结果再生成 PDF。**

这个升级对实际可用性提升最大。
